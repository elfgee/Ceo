## 목표 및 전제조건

### 🎯 진짜 목표

#### 우리가 원하는 것
```
❌ "디자이너가 코드를 잘 짜게"
✅ "디자인 QA 핑퐁 제거 → UX 고민 시간 확보"
```

#### 현재 문제점
```
Figma 디자인 (1주)
  ↓
개발팀 구현 (1주)
  ↓
QA: "이거 색상 다름" (2일)
  ↓
수정 (1일)
  ↓
QA: "간격도 다름" (2일)
  ↓
수정 (1일)
  ↓
= 총 3주 소요
= 불필요한 커뮤니케이션 비용
= PD 대기 시간 과다
= 스트레스 증가
```

#### 원하는 모습
```
Figma 디자인 (1주)
  ↓
AI로 코드 생성 (30분)
  ↓
프리뷰 확인, 완벽 (10분)
  ↓
개발팀: 바로 사용 (감사합니다!)
  ↓
= 총 1주 소요
= 남은 2주는 더 나은 UX 고민
= 더 많은 프로젝트 진행 가능
```

### ⚡ 절대 전제조건

#### CRITICAL: 코드는 빠르고 쉬워야 함
```
✅ 허용되는 것:
- AI가 30분 안에 생성
- Cursor에게 "수정해줘" 5분
- 프리뷰 확인 10분
- Git push 5분

❌ 절대 금지:
- 내가 2시간 디버깅
- Git conflict 해결 1시간
- 에러 찾느라 반나절

→ 느리거나 어려우면 즉시 중단
```

#### 시간 배분 원칙
```
PD 시간 = 100%

목표 배분:
- UX 작업: 80%
  (리서치, 디자인, 프로토타입, 테스트)
- 코드 작업: <20%
  (Cursor 프롬프트, 확인, push)

만약 코드 작업 >20%면 실패
→ UX 시간을 잃는 것
→ 본질 상실
```

## 빠른 코드 생성 실현 조건

### 1. 완벽한 Figma → Code 파이프라인

#### 목표: 설명하지 않고 코드 생성
```
❌ 현재 (느림):
디자이너: "Cursor야, 버튼은 왼쪽에 있고, 색상은..."
Cursor: "네, 만들어볼게요"
→ 설명에 30분

✅ 목표 (빠름):
디자이너: [Figma Dev Mode 코드 붙여넣기]
디자이너: "이걸 우리 토큰으로 변환해줘"
Cursor: "완료!"
→ 5분
```

#### 구체적 셋업
```
1. Figma Tokens Plugin 설치
   - Tokens Studio for Figma
   - 토큰 → JSON export
   - GitHub 자동 sync

2. Figma Dev Mode 활용
   - 컴포넌트 선택
   - "Copy as" → Code
   - Cursor에 붙여넣기

3. Cursor 프롬프트 템플릿
   - 자주 쓰는 명령어 저장
   - 복사-붙여넣기로 재사용
```

### 2. "5분 룰" - 엄격한 시간 제한

#### 원칙: 5분 안에 안 되면 중단
```
작업별 제한 시간:

✅ Figma → Cursor 프롬프트: 2분
✅ AI 코드 생성 대기: 1분
✅ 로컬 확인: 2분
합계: 5분 이내

만약 이것보다 오래 걸리면:
❌ 복잡한 디버깅: 5분+
❌ Git 문제: 10분+
❌ 빌드 에러: 20분+
→ 즉시 중단, 개발팀 요청
```

#### 실천 방법
```
정말로 타이머 사용:

1. "Button 컴포넌트 만들기" 시작
2. 타이머 5분 시작
3. Cursor에게 요청
4. 결과:
   - 5분 안에 완성 ✅ → 계속
   - 5분 넘음 ❌ → "개발팀 도와주세요"

예외 없음!
"조금만 더"는 금지
```

### 3. 템플릿화 - 반복 작업 0분으로

#### 처음 vs 두 번째
```
Button 컴포넌트 (첫 번째):
- Cursor와 시행착오: 30분
- 수정 요청 여러 번
- 템플릿으로 저장 ✅

Button 변형 (두 번째):
- 저장된 템플릿 복사
- 약간만 수정 요청
- 완료: 5분 ⚡

Card, Table, Input... 반복
→ 점점 빨라짐
→ 결국 1분 이내
```

#### Cursor 프롬프트 템플릿 예시
```
~/cursor-templates/button.md

"""
Button 컴포넌트를 만들어줘.

파일: components/ui/button.tsx
참조: @tokens/colors/colors.css
참조: @tokens/spacing/spacing.css

요구사항:
- TypeScript strict mode
- variants: default, destructive, secondary
- sizes: small, default, large  
- shadcn/ui 스타일
- Props type export
- forwardRef 사용

프리뷰 페이지도 만들어줘:
- 파일: app/button/page.tsx
- 모든 variants × sizes 조합 표시
- 사용 코드 예시 포함
"""

→ 저장해두고 재사용
```

#### 템플릿 라이브러리 구축
```
/cursor-templates/
├── button.md
├── card.md
├── input.md
├── table.md
├── form.md
└── layout.md

처음: 각 30분
이후: 각 5분
6개월 후: 각 1분
```

### 4. 개발팀 "Safety Net"

#### SLA (Service Level Agreement)
```
PD가 막히면 개발팀이 즉시 도와줌

Slack #design-system-help 채널:

긴급 (빌드 실패, 에러):
→ 30분 이내 응답

중간 (기능 작동 안 함):
→ 2시간 이내 응답  

낮음 (궁금한 점):
→ 당일 이내 응답

→ PD는 절대 혼자 고군분투 안 함
→ 5분 넘게 막히면 바로 질문
```

#### 예시 시나리오
```
PD: "npm run dev 했는데 에러나요 😭"
     [스크린샷 첨부]
     "5분째 안 풀려요"

개발팀 (10분 후):
     "node_modules 삭제하고
     npm install 다시 해보세요"

또는:

개발팀 (10분 후):
     "이건 복잡한 거네요
     제가 직접 해드릴게요"

→ PD는 30분 이상 막히지 않음
```

### 5. 자동화 & 도구

#### 필수 자동화
```
✅ ESLint (자동 코드 검사)
   - 실수 자동 발견
   - 저장하면 자동 수정

✅ Prettier (자동 포맷팅)
   - 코드 정리 자동화
   - 일관성 유지

✅ TypeScript (타입 체크)
   - Props 실수 방지
   - 자동 완성 지원

✅ Vercel (자동 배포)
   - Push하면 자동 배포
   - 1-2분 내 완료

→ PD는 "코드 생성"만 집중
→ 나머지는 자동
```

#### Hot Reload
```
npm run dev 실행 상태에서:

1. Cursor로 코드 수정
2. 저장 (Cmd+S)
3. 브라우저 자동 새로고침
4. 즉시 확인 ⚡

→ 수정 → 확인 사이클 5초
```

#### 도전 과제
```
이번 주 목표:
✅ 혼자서 할 수 있는가?

✅ 복잡해져도 여전히 빠른가?

예상 이슈:
- 조건부 스타일링 복잡
- 반응형 breakpoint
```
